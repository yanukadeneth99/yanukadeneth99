# HELP — Fork & Setup Guide

This profile README is a small system of self-hosted SVG generators plus three
live widgets that pull data from external services. Forking it and getting
everything green takes about 30–45 minutes of focused setup.

This guide is written step-by-step. Skip any section whose feature you don't
want (e.g. you can use this README without ever setting up the X integration).

---

There are five things that come from external services:
- **Visitor counter** (komarev.com) — no setup needed
- **Discord presence** (Lanyard) — see section 3
- **Spotify now-playing** (your self-hosted novatorem fork on Vercel) — see section 4
- **Pac-Man contribution graph** (auto-generated by `pacman.yml`) — no setup needed
- **Top X posts** (X API, costs ~5¢/month) — optional, see section 5

## Section 1 — Fork & rename

1. **Fork this repository** on GitHub.
2. **Rename it to match your username.** GitHub uses the special `username/username` repo as your profile README. So if your username is `octocat`, your fork must be at `github.com/octocat/octocat`.
3. **Clone your fork locally:**
   ```bash
   git clone git@github.com:<your-username>/<your-username>.git
   cd <your-username>
   ```

## Section 2 — Personalize the README

Open `README.md` and do a find-and-replace of the original owner's username with yours. There are several occurrences (badge URLs, raw asset URLs, profile links). On macOS/Linux:

```bash
# Replace <OWNER> with the original owner's username (the one this was forked from)
# Replace <YOU> with yours.
sed -i '' "s/<OWNER>/<YOU>/g" README.md
```

Or do it in your editor. The same swap is needed in:
- `.github/scripts/update-projects.mjs` — the `raw.githubusercontent.com/...` URLs
- `.github/scripts/update-x-posts.mjs` — same
- `.github/scripts/update-stats.mjs` — same

Other things in `README.md` to update:
- The hero text (`text=YOUR_BRAND`) in the `capsule-render.vercel.app` URL
- The typing tagline `lines=` query — change to your own lines (URL-encode them)
- The portfolio link (`href="https://your-portfolio.com"`)
- The Ko-fi link (`href="https://ko-fi.com/<your-handle>"`)
- The Spotify artist link in the Right Now table (or remove if you don't have one)

### Change the brand accent color

The accent purple is `#A371F7`. To switch to your own brand colour, find-and-replace `A371F7` across:
- `README.md`
- `.github/scripts/update-stats.mjs` (the `ACCENT` constant)
- `.github/scripts/update-projects.mjs` (the `STATUS.shipped.color` field if you want shipped bricks recoloured)
- `.github/scripts/update-x-posts.mjs` (the `COLOR.accent` constant — used for mention/hashtag highlighting)

The `Active` status uses `#F85149` (red) and `Upcoming` uses `#7D8590` (slate). Those are tunable in `update-projects.mjs`.

## Section 3 — Discord presence (optional)

The Discord widget uses **Lanyard** (`api.lanyard.rest`), a free open-source bot that exposes your Discord presence as a JSON API. The widget itself is rendered by `lanyard.cnrad.dev` from that data.

### Steps

1. **Join Lanyard's Discord server.** Lanyard can only read your presence if you share a server with its bot. Use the canonical invite: `https://discord.com/invite/lanyard`. (If `discord.gg/lanyard` is flaky, this longer URL works.)
2. **Enable Developer Mode in Discord.** Settings → Advanced → toggle Developer Mode on.
3. **Copy your numeric Discord user ID.** Right-click your name anywhere in Discord → "Copy User ID". It's a 17–19 digit number.
4. **Replace the placeholder in `README.md`.** Find both occurrences of the existing Discord ID (it appears in the `href` and the `src` of the Lanyard image) and replace with your own.

### Customizing

The `fallbackStatusText=Chilling..` query param in the Lanyard URL is what the widget shows when you're offline. Change to whatever you want.

## Section 4 — Spotify now-playing widget

This widget renders the song you're currently listening to (or a Spotify-branded "Currently chilling" card when you're not). It's a self-hosted **novatorem** fork on Vercel.

Why self-host? The public novatorem instance doesn't exist; novatorem is meant to be forked per-user (each instance is bound to one Spotify account). And edge-caching is needed to protect against refresh storms.

### Steps

#### 4a. Spotify Developer setup

1. Go to `https://developer.spotify.com/dashboard` → log in → **Create app**.
2. Settings → Redirect URIs → add `http://127.0.0.1:3000/api/callback`. (Spotify rejects `localhost` since 2024; use the loopback IP literal.)
3. Save `Client ID` and `Client Secret`. You'll need both.

#### 4b. Generate a refresh token (one-time)

In your browser, open this URL (substitute your `CLIENT_ID`):

```
https://accounts.spotify.com/authorize?client_id=CLIENT_ID&response_type=code&redirect_uri=http%3A%2F%2F127.0.0.1%3A3000%2Fapi%2Fcallback&scope=user-read-currently-playing%20user-read-recently-played
```

After approving, the browser will redirect to `http://127.0.0.1:3000/api/callback?code=AQ...`. The page will fail to load — that's fine. **Copy the `code=...` value from the URL bar.**

Now exchange that code for a refresh token in your terminal:

```bash
curl -X POST https://accounts.spotify.com/api/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=THE_CODE_FROM_URL" \
  -d "redirect_uri=http://127.0.0.1:3000/api/callback" \
  -u "CLIENT_ID:CLIENT_SECRET"
```

The response contains a `refresh_token` field. Save it — it does not expire.

#### 4c. Fork novatorem and deploy

1. Fork `https://github.com/novatorem/novatorem` to your account.
2. **Copy the customizations from the original fork** (the not-playing promo card with purple border, gradient sweep, and Spotify CTA pill). Specifically:
   - `api/spotify.py` — adds `get_artist(artist_id)` function
   - `api/orchestrator.py` — adds `make_promo_svg()` and branches into it when `is_playing == False`. Also bumps the response Cache-Control to `public, s-maxage=30, stale-while-revalidate=60` to absorb refresh bursts.
3. Push your novatorem fork.
4. Go to `https://vercel.com/new` → import your novatorem fork.
5. **Framework preset:** Other.
6. **Environment Variables** (all required for the basic widget):
   - `SPOTIFY_CLIENT_ID` = your Client ID
   - `SPOTIFY_SECRET_ID` = your Client Secret *(note: novatorem uses `SECRET_ID`, not `CLIENT_SECRET` — easy to miss)*
   - `SPOTIFY_REFRESH_TOKEN` = the token from step 4b
7. **Optional environment variables** (for the not-playing promo card with your artist photo + name):
   - `PROMO_ARTIST_ID` = your Spotify artist ID. Get it from your Spotify artist URL (`open.spotify.com/artist/<THIS_PART>`).
   - `PROMO_ARTIST_NAME` = display name override. Defaults to "YASHURA" — change this even if you don't set `PROMO_ARTIST_ID`, otherwise the fallback will read someone else's brand.
8. Deploy. Copy the production URL (e.g. `https://yourproject.vercel.app`).
9. **Update `README.md`** — replace the existing Spotify widget `src` URL with `https://yourproject.vercel.app/api/spotify?cb=1`. The `?cb=` parameter is a cache-buster (see Note 1 below).

#### 4d. Test

Open `https://yourproject.vercel.app/api/spotify` in a browser while playing music on Spotify. You should see an animated SVG with the track + artist + bar visualization. Stop playback and reload — you should see the promo card with your artist photo + name (if you set the optional env vars) or a green play button (if you didn't).

## Section 5 — Top X posts (optional, ~5¢/month)

The X API has been paid-only since 2026 under a pay-per-use model. The "Owned Reads" rate is **$0.001 per post returned**. With daily polling pulling 50 posts max, expect ~$0.05/month. X dedupes identical post IDs within a 24-hour window, so total monthly cost rarely exceeds $0.10 even if you tweet a lot.

If you don't want to set this up, **just delete the Top Posts on X section from `README.md` and remove `.github/workflows/update-x-posts.yml` + `.github/scripts/update-x-posts.mjs`**. The rest of the system works without it.

### Steps

1. Go to `https://console.x.com` → sign in → **Create Project + App**.
2. Top up at least $5 of credits under Billing (lasts years for this workload).
3. **Keys and tokens** → generate a **Bearer Token**. Save it.
4. **Get your numeric X user ID:**
   ```bash
   curl "https://api.x.com/2/users/by/username/<your-handle>" \
     -H "Authorization: Bearer YOUR_BEARER_TOKEN"
   ```
   The response's `data.id` field is your numeric ID. (Sites like `tweeterid.com` work too but are sometimes broken; the API call costs ~$0.001 and always works.)
5. **Add GitHub repo secrets** (see section 7).

## Section 6 — Customize your projects

The projects list (bricks under the "Projects" heading) is driven entirely by `.github/data/projects.json`. Schema:

```json
{
  "year": 2026,
  "projects": [
    {
      "name": "Project Name",
      "status": "shipped",
      "url": "https://example.com",
      "logo": "https://example.com/logo.png"
    }
  ]
}
```

| Field | Required | Notes |
|---|---|---|
| `name` | yes | Brick label (auto-wraps to 2 lines at ~13 chars each) |
| `status` | yes | One of `shipped`, `active`, `upcoming`. Determines color + fill level (100% / 25% / 0%). |
| `url` | no | Where the brick links to. Falls back to `#` if omitted. |
| `logo` | no | URL of a PNG/SVG/JPG. If fetch fails or is omitted, falls back to a colored monogram of the first letter. |

Edit the file, commit, push. The `Update Projects Section` workflow runs automatically and regenerates all bricks + counts within ~60 seconds.

## Section 7 — GitHub repo secrets

Repo → **Settings → Secrets and variables → Actions → New repository secret**. Add:

| Name | Required for | Value |
|---|---|---|
| `X_BEARER_TOKEN` | X posts | Bearer token from console.x.com |
| `X_USER_ID` | X posts | Numeric ID from section 5 step 4 |

`GITHUB_TOKEN` is auto-provided by Actions — never manually add it. It's used by the projects, X, and stats scripts for committing back to the repo.

## Section 8 — First-run

1. **Commit and push** all your changes (README replacements, projects.json edits, etc.).
2. **Trigger workflows manually** so the assets get generated immediately rather than waiting for the daily schedule. Repo → Actions tab → pick each workflow → **Run workflow**:
   - **Update Projects Section** (renders bricks + counts.json)
   - **Update Stats SVGs** (renders languages.svg + contributions.svg)
   - **Update Top 3 X Posts** (renders x-posts SVGs and avatar cache) — only if you set up section 5
3. Visit `https://github.com/<your-username>` and verify everything renders.

## Section 9 — Day-to-day

### Adding a new project

Edit `.github/data/projects.json`, add an entry, push. Workflow auto-regenerates within ~60 seconds.

### Updating the typing tagline

Edit the `readme-typing-svg.demolab.com` URL in `README.md`. The `lines=` query param is `;`-separated, URL-encoded.

### Changing a Spotify track to show

You can't directly — the widget reflects what you're actively playing on Spotify. The fallback promo card appears whenever nothing is playing.

### When something looks stale

GitHub uses a CDN image proxy called **camo** that caches profile-README images aggressively. The two ways to bust it:
- **Wait** — eventually expires (sometimes hours)
- **Change the URL** — append a query param like `?cb=N` and bump `N`. The relevant URLs are in `README.md`. The scripts already bump cache-busters automatically on regenerated assets.

### When a workflow fails

Most failures are race conditions on `git push` (two workflows trying to push at once). The workflows have built-in `git pull --rebase` + concurrency groups, but if a hard failure happens:
- Open the workflow run → see which step failed
- Re-run the workflow (Actions tab → workflow → ⋯ → "Re-run failed jobs")
- If it's persistent, check repo permissions: `Settings → Actions → General → Workflow permissions → Read and write permissions`

## Notes

### Note 1 — Cache-busters

You'll see `?cb=1`, `?cb=2`, etc. on various URLs in `README.md`. These are **manual cache-busters** for GitHub's image proxy. When you change something that should produce a visibly different image but the URL stays the same (e.g. you redeploy your novatorem with a new design but the URL is still `/api/spotify`), bump the number. The auto-generated scripts use `?v=<timestamp>` for the same purpose on assets they produce.

### Note 2 — Why so many self-hosted SVGs

Many popular profile README tools (`github-readme-stats`, `github-profile-summary-cards`) run on Vercel's free tier, which sometimes goes down or returns `DEPLOYMENT_PAUSED` (HTTP 503). The self-hosted scripts in this repo (`update-stats.mjs`, `update-projects.mjs`, `update-x-posts.mjs`) generate equivalent SVGs as committed files in your own repo. Result: nothing breaks when a third-party service goes down.

### Note 3 — Costs

| Service | Cost | Notes |
|---|---|---|
| GitHub Actions | Free | 2000 minutes/month for public repos |
| Vercel (novatorem) | Free | Hobby tier covers thousands of widget views/day |
| Spotify API | Free | App+refresh-token flow has no charge |
| Discord / Lanyard | Free | |
| Shields.io / Komarev / Capsule-Render | Free | |
| **X API** | **~$0.05–0.10/month** | Only paid service. Skip section 5 if you want zero cost. |
| **TOTAL** | **~$0.10/month** (or free if you skip X) | |

### Note 4 — Workflows that auto-commit

`update-projects.yml`, `update-x-posts.yml`, `update-stats.yml`, and `pacman.yml` all push commits back to your `main` branch. They share a `concurrency` group called `readme-writer` so they queue serially instead of racing. If you see a lot of "chore: refresh X" commits in your history, that's normal — those are the automation runs.

### Note 5 — Excluding "noise" languages

The languages donut excludes Jupyter Notebook, HTML, CSS, Markdown, MDX, TeX, Roff, Text, and reStructuredText by default — these have byte counts that inflate massively due to embedded outputs or template scaffolding, and they don't represent authorship. To re-include or exclude others, edit the `EXCLUDE_LANGS` set at the top of `.github/scripts/update-stats.mjs`.

## Troubleshooting

### Spotify widget shows the wrong (or no) artist name

- Did you set `PROMO_ARTIST_ID` in Vercel? Without it, the widget falls back to `PROMO_ARTIST_NAME` (default "YASHURA") with no photo.
- After adding the env var, you need to **redeploy** in Vercel. Env vars don't apply retroactively.
- Then bump the `?cb=` cache-buster in `README.md` so GitHub camo re-fetches.

### Discord widget 404s

- Did you actually join `discord.com/invite/lanyard`? Lanyard's bot needs to share a server with you.
- Is your Discord user ID the **numeric** one (17–19 digits)? Not your username.

### Top languages donut is empty or wrong

- Trigger **Update Stats SVGs** manually. The chart only regenerates daily otherwise.
- If most of your code is in private repos, the donut will look sparse — the GitHub API only returns public repo language stats with a default `GITHUB_TOKEN`. You'd need a PAT with `repo` scope to include private. (Adding a PAT is out of scope for this guide.)

### X cards look huge

The cards intentionally use `width="880"` (fixed), not `width="100%"`, to keep text at native CSS-pixel size regardless of monitor width. If you want them to stretch fully, change the script's `update-x-posts.mjs` block builder back to `width="100%"`. On very wide monitors that will make text appear oversized.

### Projects bricks don't update after editing projects.json

- Check that the workflow ran (Actions tab → "Update Projects Section")
- If it ran but the README didn't change, look at the job log — the script only re-writes the README between the `<!-- PROJECTS:START -->` and `<!-- PROJECTS:END -->` markers. If you accidentally deleted one of those markers, the workflow will fail.

### Pac-Man graph is broken / shows broken-image icon

The Pac-Man workflow writes its output to a separate `output` branch, not `main`. If that branch doesn't exist yet, the SVG URL 404s. Manually trigger `pacman.yml` once to create the branch and populate the SVG.

## Stripping it down

If you want a minimal version of this README — say, just the hero, badges, and projects bricks — you can safely delete:

- The Discord/Spotify table (section "Right Now")
- The X posts section + its workflow + script
- The GitHub Stats section (or just one of the two charts)
- The Pac-Man graph (delete `pacman.yml`)

Each is independent. As long as the marker pairs in `README.md` either remain intact or are removed entirely (together with their corresponding scripts/workflows), nothing breaks.
